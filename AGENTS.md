# Repository Context

## Tech Stack
- **Svelte 5** — runes mode forced via `vite.config.ts` (`runes: ({ filename }) => filename.split(/[/\\]/).includes('node_modules') ? undefined : true`)
- **SvelteKit** — adapter-auto (`.svelte-kit/` generated, gitignored)
- **Bun** — lockfile is `bun.lock`; use `bun` commands throughout
- **Better Auth** with EmailOTP flow — server side in `$lib/server/auth.ts`, client in `$lib/auth-client.ts`
- **better-sqlite3** via Kysely adapter — DB file path from `DATABASE_URL` env var, migrations via **Goose**
- Svelte MCP server configured in `opencode.jsonc` (`@sveltejs/opencode` plugin, remote MCP at `https://mcp.svelte.dev/mcp`)

## Key Commands

```sh
bun install            # install deps (also auto-run via "prepare" script)
bun dev                # start dev server
bun build              # production build
bun run preview        # preview production build
bun run check          # svelte-kit sync && svelte-check --tsconfig ./tsconfig.json
bun run check:watch    # same, with --watch
```

- Run `svelte-kit sync` before `svelte-check` if `$types` files are stale (the `check` script does this).
- Never commit `.svelte-kit/` — it's generated on install/dev.

## Environment Variables

Required: see `.env`. Key vars:
- `DATABASE_URL` — path to SQLite file (default: `storage/local.sqlite3`)
- `BASE_URL` — dev server URL
- `GOOSE_DRIVER` / `GOOSE_DBSTRING` / `GOOSE_MIGRATION_DIR` — for migrations
- `BETTER_AUTH_SECRET` — encrypt auth cookies
- `SMTP_PORT` / `SMTP_HOST` — mock SMTP for OTP testing in dev

## Architecture

- **Entrypoint:** `src/app.html`
- **Auth middleware:** `src/hooks.server.ts` — fetches Better Auth session, populates `event.locals.session` and `event.locals.user`, passes through `svelteKitHandler` for cookie/session management.
- **Auth config:** `$lib/server/auth.ts` — Better Auth instance with EmailOTP plugin (logs OTPs to console in dev) and `sveltekitCookies`.
- **Database:** `$lib/server/db.ts` — single better-sqlite3 instance.
- **Routes:**
  - `+/+layout.svelte` — root layout (Svelte 5 `$props()` runes)
  - `+/+layout.server.ts` — injects `user` into layout data from `locals.user`
  - `login/+page.svelte` — EmailOTP login: two-step flow (send code -> verify code), uses `authClient` (Better Auth Svelte client with `emailOTPClient` plugin).
- **Migrations:** `migrations/` — populated by Goose + SQL files. Trust this directory over any autogenerated schema docs.

## State Management

Always use Svelte 5 runes. This project forces runes mode — no reactive declarations (`let x = $derived()`) or stores unless importing from a library. Use `$state()` for reactive compounds, `$derived()` for computed values, `$effect()` for side effects.

## Svelte MCP

The Svelte MCP server is registered via `opencode.jsonc`. Before writing or editing `.svelte` files:
1. Always run `svelte-autofixer` to validate runes usage and Svelte 5 correctness.
2. Use `list-sections` then `get-documentation` for any SvelteKit/Svelte 5 API questions.
3. Ask before generating a playground link — never create one if code was written to project files.
